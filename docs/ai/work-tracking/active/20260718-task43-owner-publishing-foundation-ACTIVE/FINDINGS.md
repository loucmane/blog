# Task 43 Implement Owner Publishing Foundation - Findings

## Purpose
Record discoveries, gaps, risks, failed assumptions, and evidence-backed observations for Task 43.

## Findings
- 2026-07-18 - Task 42 already supplies the application-owned document validator, optimistic article versioning, immutable revisions, autosaves, lifecycle transitions, publication jobs, audit/outbox records, PostgreSQL repository, media-original port, and explicit non-production fixture backend. Task 43 should compose these contracts rather than create a parallel content model.
- 2026-07-18 - The canonical PRD and Task 43 require a protected owner shell and authentication workflow. `docs/architecture/content-persistence-foundation.md` defers authentication to “Task 59 and the owner-workflow program,” but Task 59 is an advisory cross-agent security skill, not product implementation. Task 43 therefore owns the initial app authentication foundation; Task 59 remains a later review capability.
- 2026-07-18 - Task 37 proved Tiptap OSS `3.27.3` with semantic JSON round-trip, keyboard formatting/undo, server-safe rendering, unknown-node preservation, and Axe. Registry evidence on 2026-07-18 shows `3.28.0` was released 2026-07-15; it is stable but not yet as soaked or spike-proven. Task 43 retains exact `3.27.3` and will re-run the production contract.
- 2026-07-18 - Better Auth `1.6.23` and `@better-auth/passkey` `1.6.23` remain the current stable line; `1.7.0-rc.1` is prerelease. Official Next.js 16 guidance requires server-side session validation for protected operations, and the passkey plugin provides phishing-resistant WebAuthn without a hosted identity dependency.
- 2026-07-18 - Normal owner publishing must work without developer tools, but initial production enrollment, provider credentials, and live account-recovery delivery are deployment concerns. Task 43 will provide owner-facing sign-in, passkey, and recovery boundaries with fail-closed production configuration; live enrollment and recovery drills remain Task 46 evidence.
- 2026-07-18 - The primary checkout remains owner-managed and dirty. All Task 43 mutation is isolated in `/tmp/blog-task71-closeout` on `feat/task-43-owner-publishing-foundation`.
- 2026-07-18 - The Task 42 in-memory fixture repository allowed overlapping transactions to share mutable state; a later failed transaction could replace the repository with its stale snapshot and erase another committed save. Serial transaction execution fixes the test adapter while PostgreSQL remains the production concurrency authority.
- 2026-07-18 - Next.js production bundling emitted separate domain-error class identities for the page-created process-global fixture runtime and API route handler. `instanceof` therefore converted a valid optimistic conflict into a 500. Stable internal error codes preserve classification without exposing implementation details.
- 2026-07-18 - The local quality and dependency-policy scripts spawn child processes. Sandboxed runs failed with `EPERM`, while unchanged commands passed with the required execution/network permission; this was an environment boundary, not a product or test failure.
- 2026-07-18 - `actionlint` is not installed or declared locally and no workflow file changed in Task 43. Existing workflow contract suites passed 65 policy assertions plus 46 workflow assertions; hosted protected CI remains the authoritative workflow parser boundary.
- 2026-07-18 - Independent implementation and adversarial reviews found pre-closeout gaps in v3 stored-document compatibility, stale-draft lifecycle actions, exact-revision validation, published-update scheduling, Tiptap JSON attributes, PostgreSQL revision races, per-loader authorization, production HTTP origins, mixed Tiptap minors, rollback ledger handling, revision metadata, and the weak reader/editor contract assertion. Closeout remained paused while every deterministic finding was remediated and regression-tested.
- 2026-07-18 - Scheduling now has a protected, fail-closed worker entrypoint and preserves the current published revision while an update is pending. Cancellation restores the recorded pre-schedule status, and the canonical job records the owner timezone.
- 2026-07-18 - The human screen-reader protocol required by ADR-0003 cannot be represented by Axe or synthetic browser checks. Automated keyboard, reflow, reduced-motion, mobile, focus/status, and serious/critical Axe coverage is blocking; attended assistive-technology evidence remains required before final Task 43 closeout.
- 2026-07-18 - The first full browser matrix exposed a real mobile feedback race: publication succeeded, but a delayed autosave replaced the owner-visible publication confirmation. Lifecycle feedback now has precedence over background-save feedback until the owner edits or starts another explicit recovery/save flow.
- 2026-07-18 - A second implementation review found that a lifecycle click could still race an already in-flight autosave and publish its older revision. Lifecycle actions now wait through the click-time dirty generation, abort on any failed save, and use the persisted exact revision; desktop and mobile regressions prove the published content includes the newer edit.
- 2026-07-18 - Legacy v1 portable bundles with publication jobs must verify their digest against the original wire representation before Task 43 defaults are inferred. Verifying normalized content first broke otherwise valid historical bundles; raw-wire verification plus post-verification normalization preserves integrity and backward compatibility.
- 2026-07-18 - Idempotency replay must be serialized by operation/key before a PostgreSQL mutation starts. Transaction advisory locks now make concurrent same-key requests return the same result and create exactly one revision/autosave, matching the in-memory fixture contract.
- 2026-07-18 - Programmatic revision restore initially scheduled an unnecessary autosave after the restore transaction. The editor now suppresses autosave for server/restored document loads and reports the restore lifecycle outcome directly; browser coverage proves one restored revision and persistent owner confirmation.
- 2026-07-18 - The reversible development migration has an executable down path, but rolling it back drops Task-43-added owner/auth and revision metadata. A production rollback therefore requires the Task 46 backup/restore drill or forward recovery rather than treating schema down-migration alone as lossless recovery.
- 2026-07-18 - A final blocker review proved that retaining `publishedRevisionId` after unpublish is safe only when reader visibility also consults the active publication job's `previousStatus`. Scheduled replacements now keep a reader version live only when scheduling began from `published`; unpublished/draft stories remain private until the worker completes.
- 2026-07-18 - Historical publication jobs cannot safely infer prior visibility from a retained published pointer. Legacy SQL and portable imports now infer only explicit `published` or `unpublished` article states and otherwise default to private `draft`.
- 2026-07-18 - Publication completion must discover the immutable article identity, lock the article first, then lock and revalidate the job before transitioning both in one transaction. This article→job order matches lifecycle mutations and removes the reschedule/completion lock inversion while retaining worker ownership and lease checks. PostgreSQL `FOR UPDATE` plus `SKIP LOCKED` integration coverage proves a reclaimed worker cannot be overwritten by stale completion.
- 2026-07-18 - Portable imports now validate every job's article/revision identity and require each active job to match the article's scheduled revision before any rows are written.
- 2026-07-18 - Fixture authentication is now available only under explicit `NODE_ENV=test`, loopback origin, and strong token. Playwright runs one deterministic worker against a production build started in the explicit test runtime; the normal production runtime rejects fixture mode.
- 2026-07-18 - The restore journey now saves any dirty owner work before creating the restored draft. The browser regression proves the unsaved content remains in immutable history, only one restored revision is added, and the restore control meets the 44-pixel target.
- 2026-07-18 - A later review found three deeper concurrency/integrity gaps: edits could arrive while restore was awaiting the server, a scheduled article could lack exactly one active job, and completion used the inverse row-lock order from rescheduling. Restore now disables owner mutation controls, preserves and resaves any generation that still arrives, and toggles Tiptap editability without emitting phantom updates. Scheduling and portable import require exactly one active job, and completion now locks article→job consistently.
- 2026-07-18 - Final adversarial remediation removed every fail-open scheduled-state lookup. Reader visibility, reschedule, cancellation, and unpublish now require exactly one pending/claimed job whose article, revision, and run time match the scheduled article; malformed or duplicated state stays private and requires attended recovery.
- 2026-07-18 - Owner lifecycle reliability requires more than server idempotency. The editor now freezes mutation controls while preview or lifecycle persistence drains, keeps one action key across an uncertain response, and replays that exact key so a committed publish is recovered instead of duplicated.
- 2026-07-18 - Private preview now opens only after the current dirty generation is durably saved, unpublish requires an explicit confirmation and audit reason, portable imports verify publication-pointer and schedule consistency, and one-shot toolbar controls no longer expose false toggle state through `aria-pressed`.
- 2026-07-18 - Final independent review found that completion still omitted run-time identity, orphaned jobs could not reach their terminal branch, and one poison job could abort a worker batch. Completion now verifies the sole active job and exact run time, terminally supersedes deterministic corruption, preserves claimed lease/attempt state for transient retry, and continues later jobs with explicit completed/retry/superseded reporting.
- 2026-07-18 - Final accessibility review required explicit multiline textbox semantics and complete lifecycle freezing of media controls. Both are implemented and covered in the desktop/mobile owner journeys. The final implementation and adversarial closure reviews returned `NO BLOCKERS`.
