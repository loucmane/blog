# The Integration Principle

## The Pattern We Keep Seeing
"We create good things but don't put them where they'll actually be used!"

## Root Cause
When we solve a problem or create something useful, we often:
1. Create it in isolation (work folder, memory, random location)
2. Feel satisfied that it exists
3. Move on without integrating it into the main system
4. Wonder later why things aren't improving

## The Integration Principle

### Every creation needs THREE locations:

1. **Creation Location** - Where it's built and detailed
   - Work folders for development
   - Full documentation and context
   - Complete implementation

2. **Usage Location** - Where it's actually needed
   - WORKFLOWS.md for processes
   - CLAUDE-NEW.md for quick access
   - CONVENTIONS.md for standards
   - Templates for reusable patterns

3. **Discovery Location** - Where people find it
   - Quick references in navigation
   - Links from related sections
   - Index entries
   - Cross-references

### The Integration Checklist

When creating ANYTHING useful:

- [ ] **Create it fully** in appropriate work folder
- [ ] **Identify ALL usage points** - Where will this actually be used?
- [ ] **Add to main documentation** - Don't leave it isolated
- [ ] **Create discovery paths** - How will someone find this?
- [ ] **Test the integration** - Follow the path a new user would
- [ ] **Update indexes/navigation** - Make it findable
- [ ] **Cross-reference** - Link from related areas

### Examples

**Work Tracking System**:
- ❌ Created in work folder only = Never used
- ✅ Created in work folder + Added to WORKFLOWS.md + Referenced in CLAUDE-NEW.md + Templates available

**Workflow Improvement Protocol**:
- ❌ Created as separate document = Forgotten
- ✅ Added to WORKFLOWS.md + Key principle in CLAUDE-NEW.md + Referenced in conventions

**Memory Naming Convention**:
- ❌ Decided in discussion = Not followed
- ✅ Documented in CONVENTIONS.md + Example in memory template + Reminder in workflow

### The Test
For every useful thing created, ask:
1. "Where will someone look for this?"
2. "What workflow will trigger its use?"
3. "How will they know it exists?"

If you can't answer all three, it's not properly integrated.

### Anti-Patterns to Avoid

1. **"It exists somewhere"** - If you can't quickly navigate to it, it doesn't count
2. **"I'll integrate it later"** - Later never comes
3. **"People will find it"** - They won't
4. **"It's in the work folder"** - Work folders are temporary
5. **"The memory has it"** - Memories are for context, not primary documentation

### The New Rule

**Nothing is "done" until it's integrated where it will be used.**

This means:
- Work isn't complete until documentation is updated
- Features aren't finished until workflows include them
- Patterns aren't established until they're in conventions
- Tools aren't available until they're in the navigation

### Making This Stick

1. Add to WORKFLOWS.md: "Step X: Integrate into main documentation"
2. Add to completion criteria: "Updated all usage locations"
3. Add to review checklist: "Is this findable and usable?"
4. Make integration a tracked todo item
5. Don't mark work as DONE until integrated

## The Goal

Everything useful should be:
- **Findable** when needed
- **Usable** where needed  
- **Discoverable** by those who need it
- **Maintained** as part of the main system

Not hidden in work folders, memories, or random locations!
